# Proposal: Specification-First Workflow for the Dev Harness

> **Status:** Draft for review · **Date:** 2026-06-03 · **Scope:** spa-velocity dev harness (`.ruler/` → `CLAUDE.md`/`AGENTS.md`)
> This is a proposal document only. Nothing here has been implemented yet.

---

## 1. Context — why this proposal exists

In this harness the LLM frequently jumps straight to code for features, fixes, refactors, and behavioral changes. The fallout:

- No consistent record of **why** a change was made.
- No reliable **SPEC/PRD history** for features and changes.
- No **folder convention** for documentation.
- Docs go **stale** after a bug fix or follow-up changes behavior.
- **Wrong assumptions** made during implementation are never documented, or remain uncorrected in the docs.
- Over time, **code and documentation drift apart**.

**Intended outcome.** Every *behavioral* code change must **begin** by creating or editing a Markdown SPEC before implementation, and must **end** by reconciling that SPEC with what actually shipped — verified by a dedicated drift gate and a hard CI gate. Documentation becomes part of the implementation, not an optional extra.

### How it fits the existing harness (verified by exploration)

- The harness is authored in `.ruler/` and generated by **Ruler** (`@intellectronica/ruler@0.3.40`, command `npx ruler apply`) into `CLAUDE.md` / `AGENTS.md` / `.github/copilot-instructions.md`. **Source of truth** = `.ruler/instructions.md`, `.ruler/skills/<name>/SKILL.md`, `.ruler/agents/<name>.md`.
- Existing doc system = **ADRs only**: `docs/decisions/` with `README.md` index + `_template.md`; `ADR-NNN-<kebab>.md`; append-only; status lifecycle. Governed by the `documentation-and-adrs` skill and the **layered-router principle**: *`CLAUDE.md` must never reference file paths or ADR numbers* — it references skills/agents **by name only**.
- `plan-mode` produces an **ephemeral** Step-0 (requirements + assumptions + slicing). It does **not** persist a durable artifact — the SPEC is that durable home.
- All 6 review subagents are **read-only** (Read/Grep/Glob/Bash; no Edit/Write). House style = *propose, don't write; wait for approval* (`lessons-curator`). The new `spec-steward` follows this exactly.
- Enforcement surfaces to extend: **P3.4** (force-fire skills), **P4** (review-subagent triggers), **P8 / P8.0** (output contract + definition of done), and the **Workflow Chains**.

### Decisions locked (from the clarifying questions)

| # | Decision | Choice |
|---|---|---|
| 1 | Spec location & granularity | **`docs/specs/` flat, `SPEC-NNN-<kebab>.md`** (mirrors ADRs) |
| 2 | Strictness floor | **Behavior ⇒ spec; only TDD-style waivers exempt** (non-code / type-only / config-no-behavior). "small/obvious/trivial" never valid |
| 3 | Enforcement | **Hard CI gate now** + soft `CLAUDE.md` rule + `spec-steward` subagent |
| 4 | Backfill scope | **Backfill specs for all existing features now** |

### Recommended shape (answers "skill vs agent vs pre-step vs post-step?")

**A combination of all five** — each layer does one job:

1. **A `spec-workflow` skill** — the *knowledge* layer: folder/naming, the template, create-vs-update rules, sync rules, the pre/post workflow.
2. **`.ruler/instructions.md` enforcement** — the *mandate* layer: force-fire rule (P3.4), definition of done (P8.0), workflow chains.
3. **A write-capable `spec-steward` subagent** — the *documentation owner*: the single writer of `docs/specs/**`. **Pre-implementation** it drafts/updates the SPEC from the request (the main agent presents it for approval); **post-implementation** it reconciles the SPEC with the shipped diff, applies the descriptive sync edits itself, and BLOCKs on semantic contradictions. This is the first write-capable subagent in the harness — scoped strictly to `docs/specs/` (see guardrails in §10).
4. **`architect-reviewer` as the pre-implementation plan gate (§19)** — once the SPEC is written, the **architect reviews the SPEC's design/plan/data-model** (its §5 affected areas, §6 acceptance criteria, §7 implementation plan) **before any code**, returning `APPROVE_PLAN` / `REVISE_PLAN` / `BLOCK`. The SPEC is the durable artifact it reviews (an upgrade over the old ephemeral plan). Fires per P4 (3+ files OR auth/RBAC/route-guards/state-mgmt/data-migration). REVISE/BLOCK loops back to the steward before implementation starts.
5. **The main agent orchestrates** — runs the clarification gate, delegates SPEC writes to `spec-steward`, routes the SPEC through `architect-reviewer`, presents the (architecturally-vetted) SPEC + plan for approval, writes the code/tests, then delegates the post-impl reconciliation back to `spec-steward`.

## 2. The model — SPEC vs ADR vs plan vs changelog (no overlap)

| Artifact | Answers | Lifetime | Location |
|---|---|---|---|
| **SPEC** | *What* behavior we build + *how* + acceptance criteria + assumptions | Durable; evolves with the feature | `docs/specs/SPEC-NNN-*.md` |
| **PRD** (rare) | *Why at product level* for a multi-SPEC epic | Durable | `docs/specs/PRD-NNN-*.md` |
| **ADR** | *Why* a load-bearing architectural decision + alternatives rejected | Durable, append-only | `docs/decisions/ADR-NNN-*.md` (unchanged) |
| **plan-mode** | Execution sequencing for one task | Ephemeral (persisted *into* the SPEC) | response only |
| **Change Log** | *What changed in this feature over time + why* | Embedded, append-only | a section **inside** each SPEC |

Rules of the model:
- The SPEC **cites** ADRs; it never restates rationale. If a change introduces a load-bearing decision, write an ADR via `documentation-and-adrs` and cite it from the SPEC. **SPEC ≠ ADR.**
- The Change Log is an **embedded section per SPEC**, not separate files — one source of truth per feature, so follow-ups and bug-fix behavior changes can't drift into an orphan file. (An optional repo-level `CHANGELOG.md` in *Keep a Changelog* format may be derived later for release notes; out of scope here.)
- Existing ad-hoc `docs/*-plan.md` files are superseded by SPECs going forward (left in place; not migrated unless touched).

| Kind | Pattern | Example | Notes |
|---|---|---|---|

## 5. Deliverable 3 — Mandatory PRE-coding workflow

The LLM MUST complete all steps before writing any implementation or test code:

1. **Classify the request** — feature / improvement / bug fix / refactor / behavioral change / requirement-correction / follow-up. Decide **behavioral?** If yes → SPEC required. If purely non-code, type-only, or config-with-no-behavior → state the exact waiver phrase (§8) and skip. "small/obvious/trivial" is NEVER a skip.
2. **Search for an existing SPEC** — read `docs/specs/README.md` index AND grep `docs/specs/` for the feature/behavior/affected files. This is the **anti-duplicate** step.
3. **Resolve ambiguity — the Requirements Clarification Gate (§18).** Run the ambiguity scan against the §18 checklist; classify each dimension `Known` / `Assumable-safe` / `Must-ask`. If any **Must-ask** items remain, the steward returns `NEEDS-INPUT` with batched, structured questions; the **main agent asks the user** (via `AskUserQuestion`). **Spec-writing is BLOCKED until every Must-ask is answered** (or the user explicitly downgrades it to an accepted risk). Do NOT guess past a material ambiguity.
4. **Decide create vs update** per §8 (default: if any SPEC already covers this feature, UPDATE it — never open a second).
5. **Create/update the SPEC via `spec-steward`** (the write-capable owner of `docs/specs/**`) from `_template.md`: intended behavior, context/problem, in-scope + non-goals, **numbered assumptions**, affected areas, **acceptance criteria mapped to planned tests**, implementation plan, testing plan, risks/failure modes, open questions, and a Change Log entry for this change. The main agent supplies the resolved requirements; the steward writes the file and returns its path + a summary. The SPEC must pass the **spec-readiness rubric (§18.2)** before it leaves Draft.
6. **Load-bearing check** — if the change makes a decision that would force updating 3+ skills/docs/files, write/cite an **ADR** (via `documentation-and-adrs`); the SPEC references it.
7. **Architect review of the SPEC's plan (§19).** When P4 triggers (3+ files OR auth/sessions/RBAC/route-guards/state-mgmt-rewrites/data-migration), route the written SPEC through **`architect-reviewer`**, which reviews §5 (affected areas), §6 (acceptance criteria), §7 (implementation plan) and the data model. Verdict `APPROVE_PLAN` / `REVISE_PLAN` / `BLOCK`. **`REVISE_PLAN`/`BLOCK` ⇒ the steward revises the SPEC and it is re-reviewed — no implementation begins until `APPROVE_PLAN`.** For a cross-repo change, the architect reviews **both** specs + the coordination doc (no single-side review — §16).
8. **Present the SPEC (or its diff) + the architect verdict + the plan to the user and STOP** — no implementation until the user approves. (In Claude Code this is the `plan-mode` / ExitPlanMode gate; the SPEC *is* the durable plan.)

## 6. Deliverable 4 — Mandatory POST-coding workflow

After implementation passes its tests, before declaring done / opening a PR, the main agent delegates reconciliation to **`spec-steward`**, which performs steps 1–3 and 5 by editing the SPEC itself (write-capable, scoped to `docs/specs/**`):

1. **Reconcile the SPEC with what shipped** — update *Affected areas* to match the real diff; update *Acceptance criteria* if behavior changed; ensure each AC links to its now-green test (file:line).
2. **Resolve every assumption** — mark each numbered assumption `Confirmed` or `Corrected`. A wrong assumption is **struck through and replaced inline**, with the correction noted in the Change Log. *(A `Confirmed` assumption that the code actually contradicts is NOT silently rewritten — the steward BLOCKs and surfaces it for resolution; see §10 guardrails.)*
3. **Append a Change Log entry** — `YYYY-MM-DD · PR #NN · <what changed> · <why>`; explicitly note any corrected assumption or behavior change.
4. **If an assumption was wrong** → the **main agent** also writes a `feedback`-type **memory entry** (P7 reflexive capture) and emits the curator-prompt line. *(Memory lives outside `docs/specs/`, so it stays with the main agent, not the steward.)*
5. **Flip Status** — `Draft` → `Implemented` (or `Approved` if merged later).
6. **Run the review chain** — `code-reviewer` + `qa-validator` (+ `security-reviewer` if applicable) + **`spec-steward`** (which has now applied the sync edits AND returns its verdict) + `acceptance-verifier`. `spec-steward` BLOCK or `acceptance-verifier` BLOCK is binding on "done."
7. **CI `spec-gate` must be green** (behavioral `src/**` change ⇒ paired `docs/specs/**` change present).
8. **Output per P8 contract**; the `Skills consulted:` line includes `spec-workflow`; the steward reports the exact SPEC edits it applied (for auditability).

**Mid-implementation rule (drift-prevention):** if implementation reveals the SPEC was wrong, incomplete, or based on a bad assumption, **STOP, update the SPEC first** (assumptions + Change Log), then resume. Code must never silently diverge from the SPEC.

3–8 steps (mirrors plan-mode). Each: `files:` / `tests:` / `risk:` / `slice:`.

Unresolved items blocking or shaping the work.

## 8. Deliverable 6 — Create-new vs update-existing rules

**Always search `docs/specs/` first (index + grep). One SPEC per feature/capability — never two.**

**CREATE a new SPEC when:**
- Introducing a net-new feature/capability/route/user-facing behavior with no existing SPEC.

**UPDATE the existing SPEC when (default for anything touching an existing feature):**
- Improving/extending an existing feature.
- A **bug fix that changes the intended behavior** (the SPEC records intended behavior — it must move).
- A **follow-up change** to a feature.
- A **refactor that changes observable behavior**.
- Implementation reveals a **wrong/incomplete assumption** (update assumptions + Change Log).

**SUPERSEDE (not duplicate) when:** a capability is fully replaced — mark the old SPEC `Superseded by SPEC-XXX` (don't delete), write the new SPEC with `Supersedes SPEC-NNN`. Mirrors ADR superseding.

**Multi-feature change:** update **each** affected SPEC; if it's a cross-cutting epic, create/update a `PRD-NNN` that links the child SPECs.

**Exemptions (the ONLY skips — must state the phrase, mirroring tdd-workflow waivers):**

```
SPEC waived — non-code change.
SPEC waived — type-only.
SPEC waived — config change with no behavior impact.
```

Forbidden non-waivers: "small change", "obvious fix", "trivial", "just a refactor".
Pure no-behavior refactor: no new SPEC, but if a SPEC for that feature exists, add a Change Log note (implementation detail changed, behavior unchanged).

## 9. Deliverable 7 — Documentation↔code synchronization rules

1. **Same-PR rule (mechanically enforced):** a PR whose diff touches behavioral `src/**` must include a `docs/specs/**` change in the **same PR**. The CI `spec-gate` fails otherwise.
2. **Definition-of-done rule:** "done" requires the SPEC reconciled + `spec-steward` non-BLOCK (§11).
3. **Behavior-moves-the-SPEC rule:** any change to intended behavior (incl. bug fixes) updates Section 1/6 + Change Log.
4. **Assumption-integrity rule:** no SPEC ships with an `Unconfirmed` assumption the code contradicts; corrected assumptions are struck + replaced + logged + captured to memory (P7).
5. **Affected-areas-tracks-diff rule:** `spec-steward` compares Section 5 against the actual changed files and flags mismatch as DRIFT.
6. **AC-maps-to-executed-test rule:** every AC links to a test that actually ran green (reuses the `acceptance-verifier` non-vacuity discipline).

## 10. Deliverable 8 — `spec-steward` subagent (`.ruler/agents/spec-steward.md`)

**Write-capable, scoped strictly to `docs/specs/**`** — the single owner/writer of SPEC files (first write-capable subagent in the harness). Invoked at **both** ends: pre-implementation to draft/update the SPEC, post-implementation to reconcile it with the shipped diff and verify. It *applies* descriptive sync edits itself, but **never** rewrites a SPEC to paper over a semantic contradiction — those it BLOCKs.

```markdown
---
name: spec-steward
description: Use ALWAYS for any behavioral change (feature/improvement/bug-fix/behavioral/refactor-with-behavior-change/follow-up) as the single owner and WRITER of docs/specs/**. PRE-implementation: run the requirements clarification gate (scan for ambiguity; return NEEDS-INPUT with questions when the request is underspecified — never guess past material ambiguity), then locate the governing SPEC (dedupe) and create/update it, enforcing the spec-readiness rubric. POST-implementation: reconcile the SPEC with the shipped diff, APPLY the descriptive sync edits (affected-areas, AC↔test links, change log, status), and verify. Write-capable but scoped to docs/specs/ ONLY — never edits src/, tests, config, ADRs, skills, or CLAUDE.md. Returns NEEDS-INPUT / SYNCED / UPDATED / BLOCK; flags semantic contradictions instead of auto-fixing. BLOCK is binding on "done." NOT for non-code, type-only, or config-no-behavior changes.
tools:
  - Read
  - Grep
  - Glob
  - Bash
  - Edit
  - Write
---

# Spec Steward (SPA)

The documentation owner. Where code-reviewer checks design, qa-validator checks
coverage, security-reviewer checks AuthZ, and acceptance-verifier runs the live suite —
this agent **owns the SPEC and keeps it true to the code**. It is the ONLY agent that
writes `docs/specs/**`. Runs in fresh context; willing to BLOCK.

## Write scope (hard guardrail)
You may create/edit files under `docs/specs/**` ONLY (plus the `docs/specs/README.md`
index). You MUST NOT edit `src/`, tests, config, `docs/decisions/`, `.ruler/`, skills,
or `CLAUDE.md`. You MUST NOT change code to match a spec — that's the main agent's job.
You apply DESCRIPTIVE sync (what the code now is); you never apply SEMANTIC cover-up
(rewriting intended behavior to match code that's actually wrong).

## Mode
You run in one of two modes, named by the caller:
- **PRE** — before implementation. Create or update the governing SPEC from the request.
- **POST** — after implementation. Reconcile the SPEC with the shipped diff + verify.

## Process
### 0. Required reading
- CLAUDE.md — P3 (spec-first), P4 (this gate), P8/P8.0 (done).
- `.claude/skills/spec-workflow/SKILL.md` — format, create/update rules, sync rules.
- `.claude/skills/documentation-and-adrs/SKILL.md` — ADR cross-citation; SPEC ≠ ADR.
- `docs/specs/README.md` — the index (dedupe check).
### 0.5 Discovery
List `docs/specs/`; locate the SPEC(s) whose `feature_paths` intersect the request/diff.

### PRE mode
1. Search the index + grep for an existing SPEC for this feature. If one exists → UPDATE
   it (never open a second → that's a BLOCK). Else allocate the next `SPEC-NNN`.
2. **Ambiguity scan (clarification gate, §18).** Run the §18.1 checklist; verify what the
   codebase already answers FIRST. Classify each dimension Known / Assumable-safe /
   Must-ask. If any Must-ask remain → return **`NEEDS-INPUT`** with a batched, structured
   question list and write nothing. The main agent asks the user; you are re-invoked with
   the answers.
3. Write the SPEC from `_template.md` using the resolved requirements: summary, scope +
   non-goals, numbered assumptions (low-risk leftovers as `Unconfirmed`), affected areas,
   acceptance criteria mapped to PLANNED tests, plan, testing plan, risks, open questions,
   and an opening Change Log entry. Status `Draft`. **Enforce the §18.2 readiness rubric** —
   if you cannot satisfy it from the inputs, return `NEEDS-INPUT`, not a half-spec.
4. Update `docs/specs/README.md` index. Return the SPEC path + a summary for the main
   agent to present for approval. Verdict `UPDATED`.

### POST mode
1. **Identify the governing SPEC(s).** Map changed files → SPEC via `feature_paths`.
   Zero matches for a behavioral change → BLOCK. Two SPECs for one feature → BLOCK.
2. **Reconcile + APPLY (descriptive).** Edit the SPEC so *Affected areas* matches the
   real diff; link each AC to its now-green test (file:line); append a Change Log entry;
   advance Status to `Implemented`; mark assumptions `Confirmed`/`Corrected`.
3. **Assumption integrity (semantic — DO NOT auto-fix).** If a `Confirmed` assumption
   is contradicted by the code, or an AC has no executed-green test → BLOCK and surface
   it. Never rewrite intended behavior to match wrong code.
4. **Verify.** Re-read the edited SPEC; confirm every check in the matrix passes.

### Verdict
- **NEEDS-INPUT** (PRE only) — material ambiguity unresolved; returns the question list,
  writes nothing. The main agent must ask the user before you proceed.
- **SYNCED** — SPEC already matched; no edits needed.
- **UPDATED** — SPEC created/updated (PRE) or descriptive drift found and FIXED (POST);
  edits listed below.
- **BLOCK** — semantic contradiction, missing AC test, no governing SPEC, duplicate SPEC,
  or a presented SPEC failing the §18.2 readiness rubric. Binding on "done"; needs
  main-agent/user resolution (not an auto-fix).

## Output format
​```
## Spec Sync
Verdict: SYNCED | UPDATED | BLOCK
Mode: PRE | POST
Governing SPEC(s): <paths>
Diff reviewed: <files, lines>   (POST)

### Edits I applied (docs/specs/ only)
- docs/specs/SPEC-NNN:LL — <what changed>  (e.g. "added auth.spec.ts:42 to AC2 row")

### Reconciliation matrix
| Check | Result | Evidence |
|---|---|---|
| SPEC exists & touched this change | pass/BLOCK | docs/specs/SPEC-NNN ... |
| Affected-areas matches diff | fixed/ok | <files added to §5> |
| Every AC maps to an executed-green test | fixed/BLOCK | AC2 → no test |
| No contradicted assumptions | ok/BLOCK | A3 contradicted at src/... |
| Change Log entry present | fixed/ok | — |
| No duplicate SPEC | ok/BLOCK | — |

### Needs human/main-agent resolution (BLOCK only)
- <semantic contradiction the steward refused to paper over>

### Sources read
- CLAUDE.md, spec-workflow, documentation-and-adrs, docs/specs/README.md
Confidence: 0.XX (per CLAUDE.md P8.1)
​```

## Forbidden behaviors
- Editing anything outside `docs/specs/**` (never src/, tests, config, ADRs, .ruler/,
  skills, CLAUDE.md).
- Changing code to make it match the SPEC (main agent's job).
- Rewriting a SPEC to hide a contradicted assumption or a missing AC test — BLOCK instead.
- Creating a second SPEC for a feature that already has one — update it.
- Marking a change `Implemented` when an AC has no executed-green test.
- Treating an unrun test as AC coverage (reuse acceptance-verifier's non-vacuity rule).
```

## 11. Deliverable 9 — Definition of done (extends P8.0)

A behavioral change is **not done** — no "ask the user to test," no PR — until **all** hold:

- A governing SPEC was **created or updated in the same change**, Status advanced to `Implemented`.
- The SPEC **passed the §18.2 readiness rubric** (no `TBD`/placeholder in load-bearing sections, ACs falsifiable) and **no Must-ask ambiguity shipped unresolved** — material questions were asked and answered, not guessed.
- When P4 triggered, the SPEC's plan was **`architect-reviewer`-approved (`APPROVE_PLAN`)** before implementation began (§19); any `REVISE_PLAN`/`BLOCK` was resolved in the SPEC, not worked around in code.
- **Every acceptance criterion maps to a test that actually ran green** (executed, non-vacuous — reuses P8.0 + acceptance-verifier discipline).
- **All SPEC assumptions are `Confirmed` or `Corrected`** (none `Unconfirmed`, none contradicted by code).
- The SPEC **Change Log** has an entry for this change; **Affected areas** matches the real diff.
- `spec-steward` returned **non-BLOCK**, and (for user-facing features) `acceptance-verifier` returned non-BLOCK.

## 14. Verification (end-to-end)

1. **Gate fires (negative):** edit a `src/**/*.tsx` behavioral line with **no** `docs/specs/**` change → `bash scripts/spec-gate.sh` and CI both **fail** with a clear message.
2. **Gate passes (positive):** add/modify the governing SPEC in the same diff → gate **passes**.
3. **Waiver path:** a type-only edit with commit token `[skip-spec: type-only]` → gate **passes**.
4. **Harness self-test:** `npm run test:claude` green; new acceptance assertions for P3.0/P3.4/P4/P8.0 pass.
5. **Steward auto-sync (POST):** point `spec-steward` at a sample PR where Affected-areas omits a changed file → it **applies** the edit to the SPEC and returns **UPDATED** listing the change; re-run → **SYNCED**. Then point it at a PR where the code violates a `Confirmed` assumption → it returns **BLOCK** (refuses to rewrite intended behavior) and surfaces it for resolution.
6. **Write-scope guardrail:** confirm the steward never edits outside `docs/specs/**` — give it a PR needing a code fix and verify it BLOCKs/flags rather than touching `src/`.
7. **Round-trip:** run one real small feature through PRE→code→POST; confirm SPEC created, ACs mapped to green tests, Change Log entry present, `spec-steward` SYNCED, CI green.
8. **Dedupe:** attempt a second SPEC for an already-specced feature → `spec-steward` returns **BLOCK** ("duplicate SPEC").

## 15. Open trade-offs to weigh during review

- **Cost vs. rigor.** This adds a mandatory pre-step + a 7th subagent + a CI gate. It is the right call for behavioral changes on a product with real auth/RBAC/PII surface; pair it with the **tiered-ceremony** idea from the harness review so non-behavioral edits stay cheap (the waiver phrases already do most of this).
- **Backfill effort.** "Backfill everything now" is a sizable up-front documentation project competing with delivery. Recommend running it as a tracked checklist in `SPEC-000`, time-boxed, highest-risk features (auth, RBAC, collections, chat, SQL connections) first even within the full-backfill scope.
- **CI false positives.** The behavioral-diff heuristic will occasionally flag a genuinely non-behavioral `.ts` edit; the three waiver tokens are the escape hatch. Expect a short tuning period.
- **Write-capable steward (new precedent).** `spec-steward` is the first subagent in your harness with `Edit`/`Write` — all 6 existing reviewers are read-only "propose, don't write." This is deliberate (it *is* the documentation agent), but it breaks the house convention and merges "judge" + "fix" in one actor. Two guardrails keep it safe: (1) **write scope is hard-limited to `docs/specs/**`** — it can never touch `src/`, tests, config, ADRs, or the harness; (2) it applies only **descriptive** sync (what the code now is) and **BLOCKs** on semantic contradictions (a `Confirmed` assumption the code violates, a missing AC test) rather than papering over them by rewriting intended behavior. It also reports every edit it applied, for auditability. **Per §17's own rule, write-scope containment is mechanically enforced (test #10), not left to the agent's prose** — a `spec-steward`-attributed change touching anything outside `docs/specs/**` fails CI. If even that proves too much authority in one agent, the **default fallback** is the read-only variant (steward proposes BEFORE/AFTER, main agent applies) — the rest of the design is unchanged.

## 16. Cross-repo documentation model (spa-velocity + api-velocity)

**Decision:** specs live **per-repo, split by layer**, cross-linked bilaterally (the same pattern as your existing bilateral ADRs). Documentation lives **where the change happens and where a gate can fire on it** — a spec you can't gate is a spec that drifts.

### 16.1 The split

| Layer | Repo | The spec owns | Tests that prove its ACs |
|---|---|---|---|
| **Contract / domain** | `api-velocity/docs/specs/` | Entities, relationships, endpoints, request/response DTOs, business rules, RBAC scopes, persistence/migrations | Jest unit + API e2e |
| **Experience / UI** | `spa-velocity/docs/specs/` | Screens, flows, forms, client-side validation, states, what the user sees/does | Vitest + Testing Library + Playwright |

These are **two documents about two surfaces**, not a duplicate. The dedupe rule still holds, refined: **one spec per (feature × layer)** — never two specs for the *same* layer of the same feature. A cross-cutting feature has one `contract` spec and one `ui` spec that **cross-reference** each other via the `counterpart_spec` frontmatter field.

### 16.2 Each repo is self-sufficient

Both repos get the **full kit**: `docs/specs/` + index + template, a `spec-workflow` skill, a `spec-steward` agent, and a `spec-gate` CI workflow. Consequence: a change in **either** repo fires **that repo's own** gate and steward — so a **backend-only** change (entity/relationship, no UI touched) forces `api-velocity`'s spec to update. **This closes the silent-drift gap** from the users-CRUD walkthrough.

### 16.3 How a cross-repo change flows (the users example)

```
Request: User↔Project → many-to-many; +phoneNumber, +department  (fixes bug X)

PRE                                                 ↓ both repos, before code
 api-velocity spec-steward → UPDATE api#SPEC-007    (entities, N–N rel, DTOs, RBAC, migration)
 spa-velocity spec-steward → UPDATE spa#SPEC-002    (multi-select, +2 fields, client validation)
 confirm counterpart_spec links point at each other
 write ONE coordination doc binding the paired PRs  (cross-repo-workspace Rule 3)
 architect-reviewer → reviews BOTH specs + coord doc as one plan (§19.4)  → APPROVE_PLAN
 present both spec diffs + architect verdict + plan → STOP for approval

IMPLEMENT → api PR (entity + migration + DTO) ⟷ spa PR (schema + types + form + table)

POST                                                ↓ each repo independently
 api spec-gate: src/**/*.entity.ts changed ⇒ docs/specs/** changed ✓  + migration-gate ✓
 spa spec-gate: src/** changed ⇒ docs/specs/** changed ✓
 each spec-steward POST: reconcile its spec to its diff, link ACs→green tests, UPDATED
 cross-link lint: counterpart_spec links resolve both ways ✓   (ships now)
 [DEFERRED] OpenAPI drift check: contract change ⇒ spa types regenerated  (needs ci-gates Phase C)
```

The **cross-link lint** (§17.1 #4, pure markdown) + each repo's **own spec-gate** are the install-time guarantee. The **`counterpart_spec` cross-links** are the *human* link and the **coordination doc** is the *process* link that keeps paired PRs from merging half-done. The **OpenAPI artifact** would add a *machine* contract-link — but it is **deferred**: it depends on `ci-gates-plan` Phase C (`@nestjs/swagger` emission + `npm run codegen:api`), which is unbuilt. Do not count it as an install-time guarantee.

### 16.4 Cross-link integrity (a new steward check)

Each `spec-steward` adds one check to its matrix: **if my spec's `feature_paths` consume a cross-repo contract (e.g. an API endpoint/DTO) and my `counterpart_spec` is empty or points at a non-existent spec → DRIFT** (it sets/repairs the link) **or BLOCK** (if the counterpart spec doesn't exist yet for a contract this UI clearly depends on). This is what makes "the agent notices a backend change that should move the frontend spec" mechanical rather than hopeful.

### 16.5 What goes in `cross-repo-workspace`

Per the layered-router rule, paths/numbers don't go in `CLAUDE.md`. The two-repo spec convention (this section) is codified in the **`cross-repo-workspace` skill** (both repos), alongside the existing coordination-doc and bilateral-ADR rules — as a new Rule + an ENFORCE directive the stewards honor.

### 16.6 The two stewards

One `spec-steward` per repo, **same contract, specialized by context**:

- **Same** across both: the job (own `docs/specs/`), the verdicts (`SYNCED`/`UPDATED`/`BLOCK`), the write-scope guardrail, the cross-link check (§16.4). Authored from a shared body so they can't drift apart.
- **Different** per repo: the domain knowledge comes from each repo's `spec-workflow` skill + `repo-conventions` (spa = React/Zod/Vitest/Playwright; api = NestJS/TypeORM/Jest/migrations). A small layer-specific section each: the api steward checks a migration shipped with an entity change; the spa steward checks each AC maps to a Vitest/Playwright test.
- **Boundary:** each steward **writes only its own repo's `docs/specs/`** but **reads both repos** (the session has both) to verify the `counterpart_spec` link. Reads both, writes one — the write-scope guardrail (§10) is never violated cross-repo.

## 18. Requirements clarification gate (anti-ambiguity)

**The problem this closes:** a SPEC built on unvalidated guesses looks unambiguous but is confidently wrong — the exact root cause ("the LLM made a wrong assumption") this whole workflow exists to kill. So **before** a SPEC is written, the agent must prove it understands the request well enough, and **ask the user** to resolve material ambiguity. Asking is not optional politeness — it is a gate.

### 18.1 The ambiguity scan (run in PRE, before writing)

For every request, classify each dimension below as **Known** (stated in the prompt OR discoverable in the codebase), **Assumable-safe** (a reasonable default exists, low blast-radius — record as a numbered Assumption), or **Must-ask** (multiple plausible readings, or high blast-radius, or hard to reverse):

| Dimension | Example "Must-ask" trigger |
|---|---|
| **Goal / problem / target user (role)** | "improve users page" — improve *what*, for *which* role? |
| **Scope boundaries (in / out)** | unclear whether a related feature is included |
| **Behavior — happy path + named edge/error cases** | what happens on conflict, empty, unauthorized, partial failure? |
| **Data model / contract** (api) | field types, **cardinality**, nullability, validation, relationship direction |
| **RBAC** | which roles/scopes may perform this? |
| **Acceptance criteria** | how do we *prove* it's done — falsifiable? |
| **Affected surfaces** | which features/routes/endpoints, one repo or both? |
| **UX states** (spa) | loading / empty / error / success copy + validation messages |
| **Non-functional** | perf budget, security constraint, backward-compat requirement |

### 18.2 Spec-readiness rubric (falsifiable — "enough understanding" is not a vibe)

A SPEC may NOT leave `Draft` / be presented until **all** pass:
- Goal + target user stated.
- In-scope **and** out-of-scope explicit.
- Every acceptance criterion is falsifiable and testable (no "works correctly").
- Data model / contract fully typed — no "etc.", no unspecified type/cardinality/nullability.
- RBAC stated (or explicitly "no RBAC impact").
- Every named edge/error case has defined behavior.
- **No `TBD` / `???` / `<placeholder>` in any load-bearing section.**

Whatever remains genuinely uncertain but low-risk after the gate becomes a numbered `Unconfirmed` Assumption — still surfaced at the §5 step 7 approval, never silently buried.

### 18.3 Anti-over-asking guard (respect the user's time)

The harness already values *not* asking what it can verify. So:
- **NEVER ask** what the codebase answers (existing patterns, types, conventions) — **verify first** (grep/read), then ask only what's left.
- **NEVER ask** trivia with a clear convention default — apply the default, record it as an Assumption, move on.
- **DO ask** when: multiple plausible *product* behaviors; data-model cardinality/nullability/validation the code can't reveal; RBAC intent; what counts as "done"; anything high-blast-radius or hard to reverse.
- **Batch** into one structured round (`AskUserQuestion`, ≤4 at a time), with options where possible — don't drip-feed.

### 18.4 SPEC vs PRD — scale the questioning to the change

- **SPEC** (default, single feature/behavior/layer): clarify behavior + contract + ACs.
- **PRD** (epic spanning ≥2 SPECs, or framed as a product initiative): clarify *additionally* — problem framing, target persona, **success metrics**, scope across child SPECs, phasing/rollout. A PRD that can't state a success metric isn't ready.
- The agent picks SPEC vs PRD up front and asks the matching depth of questions.

### 18.5 Orchestration (who asks whom)

Subagents run autonomously and can't hold an interactive loop with you — so the **main agent owns the questioning**:

```
main agent → spec-steward (PRE, scan)        → returns NEEDS-INPUT + structured question list
main agent → AskUserQuestion (batched)       → user answers
main agent → spec-steward (PRE, write)       → writes the now-unambiguous SPEC, returns it
main agent → present SPEC + plan, STOP        → approval gate (§5.7)
```

This adds a fifth PRE verdict to the steward: **`NEEDS-INPUT`** (questions for the user), alongside `SYNCED` / `UPDATED` / `BLOCK`.

### 18.6 Deterministic backstop

"Did the agent ask enough?" is judgment (smoke-tested per §17.2). But "the resulting SPEC has no unfilled required fields" is mechanical: a **spec-completeness lint** (§17.1, test #9) fails any presented/`Implemented` SPEC that has an empty required section or a placeholder token (`TBD`/`???`/`<...>`). So even if the agent under-asks, an incomplete SPEC can't pass the gate silently.

## 19. Architect review — the pre-implementation plan gate

The SPEC is not just documentation; it is the **plan**. So before any code, the SPEC's *technical design* gets an independent review by **`architect-reviewer`** — your existing PRE-implementation subagent. This is a deliberate upgrade: the architect now reviews a **durable, versioned artifact** (the SPEC) instead of an ephemeral plan that vanishes after the turn.

### 19.1 When it fires

Per your P4 trigger matrix — when the SPEC's plan involves **3+ files OR auth/sessions/RBAC/route-guards/state-management-rewrites/data-migration**. Smaller, low-risk specs skip it (the clarification gate + readiness rubric + user approval suffice). It is **PRE-implementation only** — the POST review chain (code-reviewer / qa-validator / security-reviewer / spec-steward / acceptance-verifier) is separate and unchanged.

### 19.2 What it reviews (and what it does NOT)

| Reviews (SPEC sections) | Against |
|---|---|
| §1 intended behavior, §3 scope | coherence, scope discipline (no creep) |
| §4 assumptions | are any architecturally risky / unvalidated? |
| §5 affected areas + data model | layering, coupling, the right module boundaries, migration safety |
| §6 acceptance criteria | are they architecturally complete (failure modes, boundaries)? |
| §7 implementation plan + slicing | SOLID/DRY/KISS/SoC/YAGNI, repo conventions, simpler alternative? |

It does **not** do doc-sync (that's `spec-steward`), coverage taxonomy (`qa-validator`), or security depth (`security-reviewer`) — anti-overlap holds. Verdict: **`APPROVE_PLAN` / `REVISE_PLAN` / `BLOCK`**.

### 19.3 The loop

```
spec-steward writes SPEC (passes readiness rubric)
        ↓
architect-reviewer reviews the SPEC's design/plan
   ├─ APPROVE_PLAN → present SPEC + verdict to user (§5 step 8) → implement
   ├─ REVISE_PLAN  → spec-steward revises the named SPEC sections → re-review
   └─ BLOCK        → fundamental rethink; back to clarification/design, not code
```

No implementation starts until `APPROVE_PLAN`. The architect's verdict is presented *with* the SPEC at the approval gate, so you see "spec + architect signed off on the plan" before you say go.

### 19.4 Cross-repo (bilateral, no single-side review)

For a change spanning both repos, `architect-reviewer` reviews **both** the contract SPEC (api-velocity) **and** the UI SPEC (spa-velocity) **plus** the coordination doc — together, as one plan. A single-side architect review of a cross-repo change is itself a finding (the `cross-repo-workspace` ENFORCE rule). This is what catches "the api plan and the ui plan don't actually fit together" before code, not after.
