# Chat — Natural Language Response Improvement Plan

**Status**: Draft / proposal
**Owner**: TBD
**Date**: 2026-04-08
**Scope**: spa-velocity (frontend chat UI) + api-velocity (chat module, Airweave integration, LangChain pipeline)

---

## 1. Problem Statement

The chat currently returns **raw Airweave search excerpts** (bullet lists of indexed text snippets) instead of a **natural-language, expert-style response**. When a user asks:

- *"How does the invitation flow work?"* — they want a senior engineer's explanation grounded in the code.
- *"What does the PRD say about impersonation?"* — they want a product analyst's synthesis grounded in the doc.

Instead, they see clipped excerpts and source links.

---

## 2. Root Cause (Important — read first)

The backend **already has a full LangChain + OpenAI pipeline** that generates natural-language responses grounded in Airweave results. It is wired in `api-velocity/src/modules/chat/application/services/chat-agent.service.ts`.

The relevant branching logic:

```
generateReply()  (chat-agent.service.ts:55)
 ├─ airweaveService.searchCollection()          // always runs
 ├─ if (results.length === 0) → fallback-no-results
 ├─ apiKey = configService.getOpenAiApiKey()    // line 86
 ├─ if (!apiKey)  → buildFallbackReply()   ← WE ARE HERE
 └─ else          → generateLangChainReply()    ← desired path
```

The `buildFallbackReply()` method (line 186) is what produces the "raw data" UX the user sees: top 3 excerpts as bullet points plus a sources list. It is intentionally a degraded mode for when the OpenAI key is absent.

**Verified**: `OPENAI_API_KEY` is **not set** in `api-velocity/.env`. `.env.example` does not even document it. This means every chat request falls through to the fallback.

The default system prompt (`config.service.ts:69`) is also generic:

> "You answer questions about organization knowledge bases. Use only the provided source context. Respond in structured markdown with sections ## Answer, ### Key Findings, and ### Sources. Keep attribution brief and factual."

That prompt is fine as a floor, but it does not cultivate an *expert persona*, does not distinguish **code questions** from **PRD questions** from **general questions**, and does not encourage explanation, tradeoffs, or step-by-step reasoning.

---

## 3. Goals & Non-Goals

### Goals

1. Every chat response is generated by an LLM with grounded context (RAG over Airweave).
2. Responses sound like a **senior engineer** (for code questions) or a **product expert** (for PRD / spec questions), not a search engine.
3. Responses explain **why** and **how**, not just cite excerpts.
4. For complex multi-source questions, the LLM assembles **complete context** across files, docs, and PRDs — not a single top-k slice of one query.
5. Sources remain attached for traceability (already working in the frontend).
6. Failure modes (no key, empty results, LLM error, agent error) degrade gracefully through clearly labeled fallback tiers.
7. No secrets exposed in the client bundle.

### Non-Goals

- No frontend redesign of the chat UI. `react-markdown` rendering is already in place and sufficient.
- No new vector store or re-indexing pipeline — Airweave continues to own retrieval.
- No switch away from OpenAI in Phases 0–4 (could revisit with Anthropic Claude later — see §8).
- No conversation memory summarization beyond the existing last-6-message window.
- No full state-machine orchestration (LangGraph) unless simpler phases hit a measured quality ceiling — see Phase 5.

## 5. Phased Plan

### Summary

| Phase | Name | Goal | Effort | Ship trigger |
|---|---|---|---|---|
| **0** | Unblock | Turn on the existing LangChain path (config + observability) | <1h | Immediately |
| **1** | Expert persona prompt | Responses sound like a senior engineer / product expert | ~half day | Immediately after Phase 0 |
| **2** | **Agentic RAG with LangChain `AgentExecutor`** | LLM drives multiple retrievals via tool calling; assembles whole context | ~1 day | If eval set shows Phase 1 is insufficient on multi-source questions |

**Recommended critical path** for the user's "professional, natural-language, expert answers" goal: **0 → 1 → eval set → 2**. Phase 3 runs in parallel as needed. Phases 4 and 5 are conditional escape hatches.

Each phase ships as its own PR with its own tests. No bundling.

### Phase 1 — Expert Persona System Prompt

**Goal**: the response *feels* like an expert, not a search-result summarizer.

1. Replace the default prompt in `config.service.ts:69-74` (or, preferred: keep the default broad and override in `.env` via `CHAT_SYSTEM_PROMPT`) with an expert persona prompt. Draft:

   > You are a senior software engineer and product analyst with 15+ years of experience. You answer questions about a specific organization's codebase, documentation, and product requirements, grounded **only** in the source context provided below. Your audience is another experienced engineer or product manager.
   >
   > **How to answer**:
   > - Lead with a direct, natural-language answer in 1–3 sentences.
   > - Then explain the **mechanism** or **intent**: how the code works, or what the PRD means and why.
   > - Use the source context as evidence. Quote short phrases only when precision matters. Do not paste large excerpts.
   > - When explaining code, describe the control flow, key types, side effects, and failure modes. Prefer plain prose over bullet lists unless enumerating distinct items.
   > - When explaining a PRD or spec, describe the user-facing behavior, acceptance criteria, and any constraints. Flag ambiguity.
   > - If the context is insufficient to answer confidently, say so explicitly and point to what additional source material would help. Never invent APIs, file paths, or requirements that are not in the context.
   >
   > **Format**: Markdown. Use headings only when the response is long enough to warrant them. Prefer flowing prose. Code fences for code. Always include a short "Sources" list at the end referencing the provided source names.
   >
   > **Tone**: confident, concise, specific. No hedging filler. No restating the question.

2. Keep the prompt configurable via `CHAT_SYSTEM_PROMPT` env var so we can iterate without redeploying code.
3. Add a `config.service.spec.ts` test covering the new default.
4. Add a `chat-agent.service.spec.ts` integration-style test using a mocked `ChatOpenAI` that asserts the system prompt passed in matches the configured one.

**Note on intent (code vs PRD vs general)**: the persona prompt above already tells the model to branch its explanation style based on whether the context is code or a spec. GPT-4o handles this implicitly — a standalone intent-classification step is intentionally not a separate phase. If this in-prompt approach proves uneven in practice, Phase 2's agentic tool makes intent explicit via the `entityType` filter and subsumes the need for a classifier entirely.

**Acceptance**:
- Same questions the user complained about now return flowing prose with real explanation, not bullet excerpts.
- Manual QA with 3 seed questions (one code, one PRD, one ambiguous) produces qualitatively expert-tone output.

**Effort**: ~half day.

### Phase 2 — Agentic RAG with LangChain `AgentExecutor`

**Why this phase exists**: Phase 1 is single-shot RAG — one search query, top-5 results, stuff-and-answer. It produces *competent* answers but cannot assemble the whole picture when a question spans multiple files, layers, or documents. Full LangGraph agentic RAG (see Phase 5) is powerful but heavy: a new dependency, a state machine, LangSmith-or-bust observability, ~600–900 lines across several files.

**This phase is the middle ground**: let the LLM drive **multiple** retrievals via **tool calling**, orchestrated by LangChain's built-in `AgentExecutor`. No graph. No state machine. No new top-level dependency — everything is already in the installed `@langchain/*` packages. This is where "professional-grade answers" realistically starts.

#### Core idea

Instead of the service deciding the search strategy (one query, top 5, done), the LLM gets a `search_knowledge_base(query, entityType?)` **tool** and calls it as many times as it needs. For a question like *"How does the invitation flow work?"*, the agent naturally issues 3–5 targeted searches ("invitation endpoint", "invitation email template", "invitation token schema", "invitation acceptance UI", "invitation RBAC guard"), reads the merged results, and synthesizes one coherent expert answer grounded in **all** of them.

The agent picks its own queries — often much better than the user's raw question — and stops when it has enough, or when `maxIterations` trips.

#### Architecture

```
[user question]
       │
       ▼
┌─────────────────────────────────────────────────┐
│   AgentExecutor (maxIterations: 5)              │
│                                                  │
│   LLM ──▶ "I need to search"                    │
│    │                                             │
│    ▼                                             │
│   search_knowledge_base(query, entityType?)     │
│    │                                             │
│    │  wraps airweaveService.searchCollection()  │
│    ▼                                             │
│   results ──▶ LLM reads ──▶ "need more?"        │
│                                   │              │
│                         yes ◀─────┴─────▶ no    │
│                          │                  │    │
│                          ▼                  ▼    │
│                     loop back           synthesize│
└─────────────────────────────────────────────────┘
       │
       ▼
[grounded answer + aggregated sources from all tool calls]
```

#### Tool definition (sketch)

```typescript
// chat-agent-tools.ts
import { tool } from '@langchain/core/tools';
import { z } from 'zod';
import type { AirweaveService } from '../../../airweave/application/services/airweave.service';

export function createSearchKnowledgeBaseTool(
  collectionId: string,
  airweaveService: AirweaveService,
  sourcesSink: Array<AirweaveSearchResultSummary>, // mutated; deduped later
) {
  return tool(
    async ({ query, entityType }) => {
      const response = await airweaveService.searchCollection(collectionId, {
        query,
        tier: 'classic',
        limit: 8,
        offset: 0,
      });

      const filtered = entityType
        ? response.results.filter((r) => r.entityType === entityType)
        : response.results;

      sourcesSink.push(...filtered); // aggregate across all tool calls

      // Return a compact JSON string the LLM can read
      return JSON.stringify(
        filtered.map((r) => ({
          name: r.name,
          type: r.entityType,
          source: r.sourceName,
          url: r.webUrl,
          excerpt: r.text.length > 1500 ? r.text.slice(0, 1497) + '...' : r.text,
        })),
      );
    },
    {
      name: 'search_knowledge_base',
      description:
        'Search the organization\'s indexed knowledge base (code, docs, PRDs). Call this MULTIPLE times with different, specific queries to gather complete context before answering. Prefer narrow targeted queries over broad ones. Use the optional entityType filter when the question clearly targets code vs documentation.',
      schema: z.object({
        query: z.string().describe('A specific, targeted search query'),
        entityType: z
          .enum(['file', 'function', 'class', 'module', 'doc', 'page', 'spec'])
          .optional()
          .describe('Optional filter to constrain results by type'),
      }),
    },
  );
}
```

#### Agent wiring (sketch inside `chat-agent.service.ts`)

```typescript
async generateAgentReply(
  apiKey: string,
  params: GenerateReplyParams,
): Promise<ChatReply> {
  const collectedSources: AirweaveSearchResultSummary[] = [];
  const searchTool = createSearchKnowledgeBaseTool(
    params.collectionId,
    this.airweaveService,
    collectedSources,
  );

  const prompt = ChatPromptTemplate.fromMessages([
    ['system', this.configService.getChatSystemPrompt()], // Phase 1 expert persona + tool-use instructions appended
    ['placeholder', '{chat_history}'],
    ['human', '{input}'],
    ['placeholder', '{agent_scratchpad}'], // required for tool-calling agents
  ]);

  const agent = createToolCallingAgent({
    llm: this.getOrCreateLlm(apiKey),
    tools: [searchTool],
    prompt,
  });

  const executor = new AgentExecutor({
    agent,
    tools: [searchTool],
    maxIterations: this.configService.getChatAgentMaxIterations(),
    returnIntermediateSteps: true,
    verbose: false,
  });

  const result = await executor.invoke({
    input: params.question,
    chat_history: this.buildLangChainHistory(params.previousMessages),
  });

  const uniqueSources = dedupeByEntityId(collectedSources).slice(0, 10);

  return {
    content: String(result.output).trim(),
    metadata: {
      generator: 'langchain-agent',
      sources: uniqueSources.map(this.toChatSource),
      resultCount: uniqueSources.length,
      toolCallCount: result.intermediateSteps?.length ?? 0,
    },
  };
}
```

The existing `generateReply()` method becomes a dispatcher with a **three-tier fallback chain**:

```
if (!apiKey)                      → buildFallbackReply()            // no LLM available
else                              → generateAgentReply()            // Phase 2 agentic
  catches error                   → generateLangChainReply()        // Phase 1 single-shot mid-tier
    catches error                 → buildFallbackReply()            // raw-excerpt last resort
```

Each tier logs its `generator` (`langchain-agent`, `langchain-openai`, `fallback-search-summary`) so degradation is observable in prod.

#### System prompt additions (append to the Phase 1 expert persona)

> **Tool usage protocol**:
> You have access to `search_knowledge_base`. Use it aggressively before answering any non-trivial question about this organization's code, docs, or PRDs.
>
> 1. First, silently decompose the question into 2–5 specific sub-queries that together would give a complete picture.
> 2. Call `search_knowledge_base` once per sub-query. Prefer narrow, specific queries over broad ones. Use the `entityType` filter when the question clearly targets code vs documentation.
> 3. Read the results. If coverage is incomplete, issue additional searches with refined queries.
> 4. Synthesize a single coherent expert answer grounded ONLY in tool results.
> 5. If after 3–4 searches the information is still insufficient, say so explicitly and suggest what additional source material would help. Do not invent APIs, file paths, or requirements that are not in the tool results.
>
> Do not answer from memory about this specific organization's code, architecture, or requirements. The tool is your only authoritative source.

#### Sources, metadata, and observability

- **Sources aggregation**: all tool calls push into a per-request `collectedSources` array. Dedupe by `entityId`, sort by `relevanceScore`, cap at 10. Emit in `metadata.sources` (same shape as today — frontend needs zero changes).
- **`metadata.generator`** = `'langchain-agent'` on the agentic happy path.
- **`metadata.toolCallCount`** = number of tool invocations. Valuable signal — see "Red flags" below.
- **Logging**: info log per request with `{ generator, toolCallCount, sourceCount, durationMs }`. Redact the query text if it might contain sensitive user input.

#### Streaming (folds the standalone Phase 4 into this phase)

`AgentExecutor` supports `.streamEvents()` which emits structured events: tool calls starting/ending, intermediate steps, and final-answer tokens. Two options:

**Option A — simple (ship this first)**: stream only final-answer tokens over SSE. Intermediate tool calls are invisible to the user but logged server-side. Zero frontend changes beyond what already exists.

**Option B — rich UX (defer)**: forward tool-call events to the frontend as new SSE event types (`thinking`, `searching`, `found`) so the UI can render a live "Searching for X..." indicator. Makes the longer latency feel professional instead of slow. Requires small additions to [chatService.ts](../src/features/Chat/services/chatService.ts) SSE parser and [ChatPage.tsx](../src/features/Chat/ChatPage.tsx) message rendering.

Recommend Option A for the first ship, Option B only if the eval set confirms Phase 2 is a keeper.

#### Red flags to monitor (and how to react)

| Signal | Likely cause | Fix |
|---|---|---|
| `toolCallCount` average = 1 | Agent not being agentic — ignoring the tool-use protocol in the prompt | Strengthen prompt, add a few-shot example of multi-query decomposition |
| `toolCallCount` average ≥ 5 (hitting `maxIterations`) | Agent thrashing, or questions genuinely too broad | Inspect traces; may need `maxIterations: 7` OR stricter "stop early" instructions |
| p95 latency > 20s | Too many sequential tool calls | Enable parallel tool calls in `ChatOpenAI` config (GPT-4o supports it); consider `gpt-4o-mini` for the agent loop and `gpt-4o` only for final synthesis |
| Repeated identical tool calls | Prompt is not telling the agent to vary queries | Add explicit instruction: "never repeat a query you have already issued" |
| Answer contradicts retrieved sources | Agent hallucinating despite grounding | Add a final self-check pass (cheap second LLM call); if insufficient, this is the trigger to graduate to Phase 5 (LangGraph + CRAG) |

**vs. Phase 1 (single-shot)**:

| | Phase 1 | Phase 2 |
|---|---|---|
| New code | ~0 lines | ~250–350 lines (2 new files + service changes) |

| Self-correction / answer grading | no | yes (CRAG / Self-RAG patterns) |

#### Acceptance criteria

- `metadata.generator === 'langchain-agent'` on the happy path for ≥ 95% of requests.
- On the eval set of 10–15 questions, Phase 2 produces **substantively better** answers than the Phase 1 baseline — defined as: better coverage (more files/docs referenced), fewer "I don't know" responses where the info does exist in Airweave, no new hallucinations.
- Average `toolCallCount` per question in the range [2, 4]. Outside that range is a red flag (see table above).
- `metadata.sources` contains unique, deduped entries aggregated from all tool calls — not just the last.
- All three fallback tiers still work:
  - Kill OpenAI key → falls through to `fallback-search-summary`.
  - Make a tool throw → executor catches, service falls through to single-shot Phase 1 path → `generator: 'langchain-openai'`.
  - Kill Airweave → `fallback-no-results` or explicit error surfaced to UI.
- Full test suite green in both repos (`npm run test:all`).

### Phase 3 — Retrieval Quality Tuning

**Goal**: give the LLM better raw retrieval so every downstream phase can do a better job. These are **tuning knobs**, not a monolithic deliverable — pick only the ones that measurably help on the eval set. Phase 3 can run in parallel with Phase 2, or after it.

Actions (pick based on measured quality gaps, not all at once):

1. **Raise `limit`** in `chat-agent.service.ts:61` (or in the agent tool in Phase 2) from 5 to 10 and adjust the prompt context budget accordingly. Measure whether the model gets confused by more chunks vs. gains coverage. Revert if quality drops.
2. **Truncate `result.text` sensibly** when building context documents (`chat-agent.service.ts:119-131`). Right now the entire chunk is passed. If Airweave returns long chunks, this can blow the token budget and push useful results out of context. Add a per-chunk char cap (~1500 chars) with a "... [truncated]" marker. (Phase 2 already does this in the tool.)
3. **Deduplicate by `entityId`** — Airweave sometimes returns multiple overlapping chunks from the same source. Keep the highest-scoring one. (Phase 2 already does this across tool calls.)
4. **Query rewriting** (mostly obsolete if Phase 2 ships): for short questions ("invitation flow?"), do a cheap LLM call to rewrite into a richer search query before hitting Airweave. Phase 2 makes this implicit — only relevant if Phase 2 is skipped.
5. **Relevance threshold**: drop results with `relevanceScore` below a tunable floor so irrelevant context does not dilute the prompt.

All of these are tuning knobs — each must be validated against the eval set.

**Effort**: hours per knob.

### Phase 4 — Token Streaming (standalone)

**Goal**: real token-level SSE streaming of the final assistant message.

Current SSE streaming chunks the **final** assistant message into 120-char slices client-side (`chat.controller.ts:46`). This simulates streaming but the LLM response is already fully computed before the first chunk is sent — so time-to-first-token equals full generation latency. With GPT-4o this can feel slow on long answers.

Actions:

1. Switch `generateLangChainReply()` from `chain.invoke()` to `chain.stream()` and pipe tokens through the existing SSE channel as they arrive from OpenAI.
2. The controller's SSE event protocol (`start`, `chunk`, `complete`, `error`) already supports this — no frontend changes needed if the chunk event shape stays compatible.
3. Persist the full message to the database only on `complete` (current behavior).
4. Add a backend timeout + cancellation path tied to the HTTP request lifecycle so an abandoned browser request cancels the LLM call.

**⚠ If Phase 2 ships, token streaming is folded in via `executor.streamEvents()` — this phase only remains relevant if Phase 2 is skipped.**

**Effort**: ~half day.

---

### Phase 5 — Full LangGraph Agentic RAG (deferred)

**Goal**: fine-grained agentic orchestration with self-correction loops. **Only build this if the eval set shows Phase 2 has a measurable quality ceiling** (e.g., persistent hallucinations despite grounding, or systematically incomplete context on specific question classes).

Key additions over Phase 2:

- **`@langchain/langgraph`** dependency.
- **`StateGraph`** with explicit nodes: `analyze_question → plan_queries → retrieve_parallel → grade_relevance → synthesize → self_check → (loop or exit)`.
- **Self-check node (CRAG / Self-RAG pattern)**: after synthesis, a second LLM pass grades the answer against retrieved sources and either accepts, regenerates, or loops back to retrieval with refined queries.
- **Explicit parallel retrieval**: sub-queries fan out in parallel instead of sequential agent iterations.
- **LangSmith**: effectively required for debugging at this level of complexity.

**Not planning in detail** until Phase 2 ships and produces data that justifies the jump.

**Effort**: ~3–5 days + LangSmith setup.

## 7. Risks & Safety

| Risk | Mitigation |
|---|---|
| `OPENAI_API_KEY` leaked via `VITE_*` prefix | Hard rule: never prefix with `VITE_`. Key lives only in api-velocity `.env` and server secret manager. Add a pre-commit or CI grep to block `VITE_OPENAI_*`. |
| PII or secrets from indexed content sent to OpenAI | Airweave already filters what it indexes. Document this in the chat system prompt context: "Do not repeat tokens, credentials, or secret-looking strings from context." Add a lightweight regex scrub on the context string before prompt assembly. |
| LLM hallucinating code paths or PRD requirements that do not exist | Prompt enforces "grounded only in provided context". In Phase 2, the tool-use protocol reinforces this ("the tool is your only authoritative source"). Evaluation set will catch regressions. Consider adding a post-generation check that every code symbol mentioned appears in at least one source chunk (Phase 5 self-check node). |
| Latency / cost ballooning in Phase 2 | `temperature: 0.2`, `maxIterations: 5` cap, and per-tool `limit: 8` are reasonable defaults. Track per-request token counts and `toolCallCount` in logs. Add a max-tokens cap in `ChatOpenAI` config. Rate limiting already exists (`CHAT_RATE_LIMIT_*`). Consider `gpt-4o-mini` for the agent loop to reduce cost. |
| Degraded-mode regression goes unnoticed | Phase 0 adds the `generator` log + dev badge so fallback mode is observable. Consider a metric/alert in prod on `generator !== 'langchain-agent'` rate (or `langchain-openai` rate pre-Phase-2). |
| Prompt injection from indexed content ("ignore previous instructions and...") | Standard mitigations: wrap context in clear delimiters, instruct the model to treat context as data not instructions. This is in the draft prompt. Not a full defense but adequate for internal use. |
| Agent thrashes and hits `maxIterations` on every request | Monitor average `toolCallCount`; see Phase 2 red-flag table for remediation. |
